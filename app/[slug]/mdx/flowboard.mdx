            <span id="initial-ui-pitfalls" className='text-lightBeige/95 text-lg'>initial ui pitfalls</span>
            <span className='text-lighterBeige font-inter font-extralight text-[1rem]'>
                although flowboard is basically a 'tldraw wrapper', there are many small nuances that made the initial development incredibly difficult.

                the nature of this concept is genuinely more intricate than it seems. sure, parents have children, which have more and more children, like a directed graph. 
                
                <span className='text-lightBeige/95 font-thin italic'>but what if the user wants to change previous frames? and how could we get each sequential frames to be coherent when they are generated in seperate api calls?</span>

                our first main issue was with <span className='text-lightBeige/95 font-normal'>bounds</span>. tldraw shapes were not the nicest thing to work with, and we spent a long time getting the initial 16:9 frame to bind images and native user edits onto the frame. 

                <Image
                    src="/projects/flowboard/16:9.png"
                    alt="flowboard-16:9"
                    width={1200}
                    height={800}
                    className="w-full h-auto object-cover rounded mb-1!"
                    sizes="(max-width: 768px) 100vw, (max-width: 1280px) 100vw, 1600px"
                    quality={75}
                    loading="lazy"
                    decoding="async"
                />

                to fix the bounds problem, we created a <span className='text-lightBeige/95 font-normal'>parent-child relationship</span>.

                <span className="route-background text-[0.9rem]">frontend/src/components/canvas/FrameActionMenu.tsx</span>
                <div className='text-[0.65rem] md:text-[0.75rem] lg:text-xs'>
                    <pre className='line-numbers'>
                        <code className='language-ts'>{`// creates a child shape
const shape: TLShapePartial = {
    id: imageShapeId,
    type: "image",
    x,
    y,
    parentId: frame.id,  // this sets the parent
    props: {
        assetId,
        w: scaledW,
        h: scaledH,
    },
};

// deleting all children for a parent
const childIds = editor.getSortedChildIdsForParent(shapeId);
if (childIds.length > 0) {
    editor.deleteShapes(childIds);
}
`}</code>
                    </pre>
                </div>

                the key idea was that: <span className='text-lightBeige/95 font-normal'>tldraw shapes needed explicit parent-child relationships via parentId rather than positioning shapes "inside" the frame's bounds</span>.

                child shapes use relative coordinates to their parent. when you place an image at (x, y) with <span className="route-background text-[0.9rem]">parentId: frame.id</span>, those coordinates are relative to the frame's top-left corner, not the canvas.

                additionally, when frames move, children move with it automatically, and we get/delete all children with <span className="route-background text-[0.9rem]">getSortedChildIdsForParent()</span> without searching the entire canvas.

                <span className='text-lightBeige/95 font-normal'>our second issue </span> was positioning the arrows correctly.

                our ui involves arrows, which indicate order of frames.

                we initially tried to manually placed the arrows, but we eventually ran into critical issues: the ui would break when frames were moved around, and glitch when overlapping.

                we fixed this with <span className='text-lightBeige/95 font-normal'>tldraw's binding system</span>:

                <span className="route-background text-[0.9rem]">frontend/src/components/canvas/ArrowActionMenu.tsx</span>
                <div className='text-[0.65rem] md:text-[0.75rem] lg:text-xs'>
                    <pre className='line-numbers'>
                        <code className='language-ts'>{`const info = useValue(
    "arrow info",
    () => {
        const arrows = editor.getCurrentPageShapes().
            filter((s) => s.type === "arrow");
        
        return arrows.map((shape) => {
        // Get tldraw's computed bounds (not manual calculation)
        const bounds = editor.getShapePageBounds(shape.id);
        
        // Position at arrow's center
        const center = {
            x: bounds.x + bounds.w / 2,
            y: bounds.y + bounds.h / 2,
        };

        return { id: shape.id, x: center.x, y: center.y /* ... */ };
        });
    },
    [editor]
);
`}</code>
                    </pre>
                </div>

                instead of manually placing the arrow, we use <span className="route-background text-[0.9rem]">editor.getShapePageBounds(shapeId)</span> which returns tldraw's computed bounds for the arrow based on its bindings.

                when frames move â†’ bindings update arrows â†’ bounds change â†’ tldraw's <span className="route-background text-[0.9rem]">useValue</span> hook re-computes â†’ ui repositions automatically.

            <span className='text-lightBeige/95 text-lg'>redis cloud</span>
            <span className='text-lighterBeige font-inter font-extralight text-[1rem]'>
                redis acts as a <span className='text-lightBeige/95 font-normal'>temporary job queue</span> for tracking video generation progress. since veo 3 takes 20-40 seconds to generate videos, we needed fast, ephemeral storage for job state.

                redis stores three types of job states with <span className='text-lightBeige/95 font-normal'>5-minute <a href="https://en.wikipedia.org/wiki/Time_to_live" target="_blank" className="a">ttls</a></span> for automatic cleanup:

                <span className="route-background text-[0.9rem]">backend/services/job_service.py</span>
                <div className='text-[0.65rem] md:text-[0.75rem] lg:text-xs'>
                    <pre className='line-numbers'>
                        <code className='language-python'>{`# Pending: User just submitted, processing hasn't started
self.redis_client.setex(f"job:{job_id}:pending", 300, self._serialize(pending_job))

# Active: Processing with Google, stores operation name for polling
self.redis_client.setex(f"job:{job_id}", 300, self._serialize(job))

# Error: Failed during generation
self.redis_client.setex(f"job:{job_id}:error", 300, self._serialize(error_job))
`}</code>
                    </pre>
                </div>

                job data is <span className='text-lightBeige/95 font-normal'>compressed with <a href="https://en.wikipedia.org/wiki/LZMA" target="_blank" className="a">lzma</a></span> before storing to save memory, especially important for operation metadata:

                <span className="route-background text-[0.9rem]">backend/services/job_service.py</span>
                <div className='text-[0.65rem] md:text-[0.75rem] lg:text-xs'>
                    <pre className='line-numbers'>
                        <code className='language-python'>{`def _serialize(self, data: dict) -> bytes:
    """Serialize + compress data for Redis storage"""
    return lzma.compress(pickle.dumps(data))

def _deserialize(self, data: bytes) -> Optional[dict]:
    """Decompress bytes from Redis storage"""
    if not data:
        return None
    return pickle.loads(lzma.decompress(data))
`}</code>
                    </pre>
                </div>

                when status is polled, redis provides sub-millisecond lookups. once the video is done, the job is automatically deleted from redis:
