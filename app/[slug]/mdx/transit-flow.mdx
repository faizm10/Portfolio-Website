        <div className='flex flex-col text-left mt-10'>
            <span className='text-lightBeige/95 text-xl'>map page architecture</span>
            <span className='text-lighterBeige font-inter font-extralight text-[1rem]'>
                the main map lives at <span className="route-background text-[0.9rem]">/map/page.tsx</span> and is the most complex file in the codebase — 2400+ lines managing mapbox layers, draw mode, route builder, simulation state, and GTFS overlays simultaneously.

                all six modes share a single mapbox instance. switching modes mutates which layers are visible and which event listeners are active, rather than mounting and unmounting separate map components. this avoids the expensive mapbox initialization cost on every mode change.

                state coordination across modes uses a combination of <span className='text-lightBeige/95 font-normal'>react state, custom browser events, and localStorage</span>. localStorage persists the active design between page reloads, so users don't lose work if they navigate away.

                <span className='text-lightBeige/95 text-lg mt-5'>simulation engine</span>
                the simulation runs entirely client-side. the <span className="route-background text-[0.9rem]">/api/simulation</span> route serves the pre-built trip artifacts, and the browser interpolates vehicle positions between waypoints using a requestAnimationFrame loop keyed to a virtual clock.

                scrubbing the time-of-day slider doesn't re-fetch data — it updates the virtual clock and the animation loop repositions all active trips on the next frame. at peak hours, this means rendering hundreds of moving points simultaneously without any server round-trips.

                <span className='text-lightBeige/95 text-lg mt-5'>route drawing</span>
                the design mode uses <span className='text-lightBeige/95 font-normal'>@mapbox/mapbox-gl-draw</span> for placing stops and drawing line geometry. when a user finishes drawing, the route builder computes a timetable scaffold based on the line's geometry and the selected frequency or departure schedule.

                custom stops (new stations) are stored in component state and layered onto the map as a separate source, distinct from the GO Transit base layer. this lets users add stops anywhere without modifying the pre-computed GTFS data.
            </span>
        </div>

                <span className='text-lightBeige/95 text-lg mt-4'>route agent</span>
                <span className="route-background text-[0.9rem]">/api/route-agent</span> takes the user's current route geometry and stop list and returns suggestions — where to add stops, which real GO corridors to connect to, how to name the line. the agent has access to the variants index so it can reason about existing GO coverage.

                <span className='text-lightBeige/95 text-lg mt-4'>schedule optimizer</span>
                <span className="route-background text-[0.9rem]">/api/schedule-optimizer</span> takes a custom route's timetable and suggests timing improvements. it looks at the route length, the requested frequency, and the distribution of departure times to recommend a more even headway or better peak service alignment.

                both endpoints stream responses so the UI can show suggestions incrementally rather than waiting for the full completion.
