14:T32acc,<h2>Intro</h2>
<ul>
<li>User Interface : The human's view of a computer</li>
<li>Formally: The place where a person expresses intention to an artifact, and the artifact presents feedback to the person.</li>
<li>A <strong>mental model</strong> is the <em>user’s understanding of how a system works</em> based on their experience and expectations.
<ul>
<li><strong>Importance:</strong> If the mental model matches reality → fewer errors. If not → confusion.</li>
<li><strong>Example:</strong> User thinks “Save” automatically uploads to the cloud, but in reality it only saves locally.</li>
</ul>
</li>
</ul>
<p><strong>User Interface</strong> vs. <strong>User Interaction</strong></p>
<ul>
<li>Interface refers to the external presentation to the user
<ul>
<li>Controls (what user manipulates to communicate intent)</li>
<li>Feedback (what the program uses to communicate its response)</li>
</ul>
</li>
<li>Interaction refers to actions by user and system over time
<ul>
<li>interaction is a dialog with a cycle alternating between the user
manipulating controls and the system responding with feedback</li>
</ul>
</li>
</ul>
<p><strong>Graphical User Interface</strong></p>
<ul>
<li>We can assume:
<ul>
<li>a pointing device (e.g. mouse)</li>
<li>a text entry device (e.g. keyboard)</li>
<li>a high-resolution display</li>
</ul>
</li>
<li>The display contains interactive elements (e.g. widgets)</li>
<li>Users interact primarily by pointing and clicking
<ul>
<li>pointing at an object of interest (e.g. widget, image, text)</li>
<li>clicking to select, drag to move, etc.</li>
</ul>
</li>
<li>The <strong>advantage</strong> is that it emphasizes recognition over recall of available actions
<ul>
<li>Easier to discover options and experiment</li>
</ul>
</li>
<li>Uses metaphor and analogy to make interface more intuitive</li>
<li>However, the disadvantage is that:
<ul>
<li>Consumes valuable screen space, forcing information off-screen</li>
<li>Switching between keyboard and pointing device time consuming</li>
</ul>
</li>
</ul>

<p>Equality Operators</p>
<p><code>==</code> (loose equality): Compares two values after converting them to a common type.</p>
<p>Example: <code>5 == "5"</code> → true</p>
<p><code>===</code> (strict equality): Compares two values without type conversion. They must be the same type and value.</p>
<p>Example: <code>5 === "5"</code> → false</p>

<h2>TypeScript</h2>
<p><strong>What is TypeScript</strong></p>
<p><strong>TypeScript</strong> is a superset of JavaScript that adds optional static typing. It catches errors during compile time and improves developer tooling like IntelliSense, code navigation, and auto-completion.</p>
<p><strong>Type Checking:</strong> TypeScript checks variable types at compile time.</p>
<p>Example:</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="tsx" data-theme="min-light min-dark"><code data-language="tsx" data-theme="min-light min-dark" style="display: grid;"><span data-line=""><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">let</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> a </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">=</span><span style="--shiki-light:#1976D2;--shiki-dark:#F8F8F8"> 123</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">;</span></span>
<span data-line=""><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">a </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">=</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> a </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">+</span><span style="--shiki-light:#22863A;--shiki-dark:#FFAB70"> "hello"</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">; </span><span style="--shiki-light:#C2C3C5;--shiki-dark:#6B737C">// Error: string is not assignable to number</span></span>

<p><strong>Implicit Types</strong></p>
<p>TypeScript can detect the type if a value is assigned immediately.</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="tsx" data-theme="min-light min-dark"><code data-language="tsx" data-theme="min-light min-dark" style="display: grid;"><span data-line=""><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">let</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> n </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">=</span><span style="--shiki-light:#1976D2;--shiki-dark:#F8F8F8"> 123</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">; </span><span style="--shiki-light:#C2C3C5;--shiki-dark:#6B737C">// inferred as number</span></span></code></pre></figure>

<h2>Input Events</h2>
<p><strong>Event-Driven programming</strong> is a programming paradigm that bases program execution flow on events. These can represent user actions or other system actions.</p>
<p><strong>Fundamental Events:</strong> raw events directly from the browser</p>
<p><strong>Translated Events:</strong> higher-level processed events derived from fundamentals</p>
<p>Types of Event Devices:</p>
<ul>
<li>Device Input Events (keyboard, pointing via mouse)</li>
<li>Window Input Events (changes via resizing and closing)</li>
<li>Window or Widget Events (pointing, focus)</li>
<li>system events (timer)</li>
<li>application events (thread like progress, completed)</li>
</ul>
<p>Purpose of Event Queue in SimpleKit run loop: stores input events so they can be processed in order during each animation frame, ensuring smooth UI updates</p>
<p>The transformed low level input is a <em>state</em></p>
<ul>
<li>each keyboard key is either <em>up</em> or <em>down</em></li>
<li>each mouse button is either <em>up</em> or <em>down</em></li>
</ul>
<p>the windowing system generates <strong>events when the state changes:</strong></p>
<ul>
<li><em>keydown</em> when a key state changes from “up” to “down”</li>
<li><em>keyup</em> when a key state changes from “down” to “up”</li>
<li><em>mousemove</em> when X/Y value changes</li>
<li><em>mouseup/mousedown</em> when button changes start from <em>up/down</em> to <em>down/up</em> (tracks the cursor) fires immediately when the mouse button is pressed up/down</li>
</ul>
<p>key differences between keydown and mousemove is that</p>
<ul>
<li>keydown is an event that fires once when a key on the keyboard is pressed down</li>
<li>mousemove fires everytime the mouse moves slightly so it would be every small movement of the cursor</li>
</ul>

<h2><strong>Hit Testing</strong></h2>
<p><strong>Hit-Testing</strong> is the process of checking whether a user’s mouse or touch input interacts with a visual element (shape) on the screen. It’s used for detecting clicks, dragging, hovering, etc.</p>
<p><strong>Shape Model vs. Shape Image</strong></p>
<ul>
<li><strong>Shape Model</strong>: The internal structure — includes geometry (x, y, radius, width, height), stroke, fill, transformations.</li>
<li><strong>Rendering</strong>: Turns the shape model into a visual image using the graphics context.</li>
<li><strong>Shape Image</strong>: The actual pixels drawn on the screen (what the user sees).</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%209.png" alt="image.png"></p>
<p><strong>Common Shape Representations</strong></p>
<ul>
<li><strong>Rectangle</strong>: top-left <code>(x, y)</code>, width <code>w</code>, height <code>h</code></li>
<li><strong>Circle</strong>: center <code>(x, y)</code>, radius <code>r</code></li>
<li><strong>Polygon</strong>: array of corner points <code>[(x0, y0), (x1, y1), ...]</code></li>
<li><strong>Polyline</strong>: like polygon, but not closed — path of lines</li>
<li><strong>Line</strong>: from <code>(x1, y1)</code> to <code>(x2, y2)</code></li>
</ul>
<p><strong>Hit Test Paradigms</strong></p>
<ul>
<li><strong>Inside Hit-Test</strong>: Checks whether the cursor is inside the <strong>filled</strong> area of a shape.</li>
<li><strong>Edge Hit-Test</strong>: Checks whether the cursor is on the <strong>stroke</strong> (border) of a shape.</li>
</ul>

<p><strong>Hit-Testing Considerations</strong></p>
<ul>
<li>Hit-testing uses the <strong>mouse's coordinates</strong> (or touch coordinates) and checks against the geometry of shapes.</li>
<li>You typically add a <code>hitTest(mx, my)</code> method to each shape class.</li>
<li>For complex shapes like polygons, <code>isPointInPath()</code> from the Canvas API can be used.</li>
</ul>
<p><strong>Optimizing Hit-Testing</strong></p>
<ul>
<li>Hit testing can become expensive when there are many shapes on screen or when shapes are complex.</li>
<li>Strategies:
<ul>
<li><strong>Avoid square roots</strong> (use squared distances)</li>
<li><strong>Early rejection</strong> using bounding box check</li>
<li><strong>Spatial partitioning</strong> (split scene into grid cells or use a quadtree)
<ul>
<li>helps limit hit tests to only shapes within a <strong>specific region</strong></li>
</ul>
</li>
<li>Only hit-test <strong>visible</strong> or <strong>interactive</strong> shapes</li>
</ul>
</li>
</ul>
<p><strong>Canvas API Support</strong></p>
<ul>
<li><code>gc.isPointInPath(mx, my)</code>: Checks if a point is inside a path.</li>
<li><code>gc.isPointInStroke(mx, my)</code>: Checks if a point is on the border (stroke).</li>
</ul>
<p><strong>Hit-Testing in Practice</strong></p>
<ul>
<li>Used for:
<ul>
<li>Selecting or clicking shapes</li>
<li>Drag-and-drop behavior</li>
<li>Hover effects (e.g., tooltips)</li>
<li>UI widgets (buttons, sliders, etc.)</li>
</ul>
</li>
<li>Each shape can implement its own logic depending on how it should react to input.</li>
</ul>
<h2>Animation</h2>
<p><strong>Animation:</strong> is the simulation of motion through a sequence of images or drawings that change over time.</p>
<p><strong>Frames and Frame Rate</strong></p>
<ul>
<li><strong>Frame:</strong> A single still image in an animation sequence.</li>
<li><strong>Frame Rate:</strong> How many frames are shown per second (measured in FPS or Hz).
<ul>
<li>Film: ~24 FPS</li>
<li>Games/UI: ~60 FPS</li>
<li>VR: 90+ FPS</li>
</ul>
</li>
<li>Higher frame rates = smoother animations. (the higher the better)</li>
</ul>
<p>| Term | What it means |
| --- | --- |
| <strong>Frame</strong> | One single image or moment in an animation |
| <strong>Frame Rate</strong> | How many frames are shown per second (e.g., 60 fps = 60 Hz) |
| <strong>Tweening</strong> | Interpolating (filling in) frames between two "key frames" |
| <strong>Key Frame</strong> | A frame that marks a major change (e.g. start or end of motion) |
| <strong>Easing</strong> | A function that controls how the speed/feel of tweening changes |</p>
<p><strong>Simulation</strong></p>
<p>Animating by simulation means using rules and functions to update the state every frame (no start/end, it just loops). For example:</p>
<pre><code>dot.x += dx;
dot.y += dy;

</code></pre>
<p>If it hits the edge, reverse direction:</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="tsx" data-theme="min-light min-dark"><code data-language="tsx" data-theme="min-light min-dark" style="display: grid;"><span data-line=""><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">if</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> (</span><span style="--shiki-light:#1976D2;--shiki-dark:#79B8FF">dot</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">.x </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">&#x3C;</span><span style="--shiki-light:#1976D2;--shiki-dark:#79B8FF"> dot</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">.r </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">||</span><span style="--shiki-light:#1976D2;--shiki-dark:#79B8FF"> dot</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">.x </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">></span><span style="--shiki-light:#1976D2;--shiki-dark:#79B8FF"> box</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">.width </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">-</span><span style="--shiki-light:#1976D2;--shiki-dark:#79B8FF"> dot</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">.r) dx </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">*=</span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583"> -</span><span style="--shiki-light:#1976D2;--shiki-dark:#F8F8F8">1</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0">;</span></span>
<span data-line=""> </span></code></pre></figure>
<p><strong>Timers</strong></p>
<p>Timers control <em>when</em> an animation or event should happen.</p>
<ul>
<li>Needs: <code>duration</code>, <code>startTime</code>, <code>update</code> logic, and optional <code>callback</code>.</li>
<li>Use <code>skTime</code> or current time from animation callback to manage timers.</li>
<li>Example: Turn something blue for 2 seconds after a click.</li>
</ul>
<p><strong>Tweening (Interpolation)</strong></p>
<p>Tweening creates smooth transitions between two values.</p>
<ul>
<li>Interpolates between a <code>startValue</code> and <code>endValue</code> over a <code>duration</code>.</li>
<li>Uses a linear interpolation formula:</li>
</ul>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="tsx" data-theme="min-light min-dark"><code data-language="tsx" data-theme="min-light min-dark" style="display: grid;"><span data-line=""><span style="--shiki-light:#C2C3C5;--shiki-dark:#6B737C">// linear interpolation from start to end</span></span>
<span data-line=""><span style="--shiki-light:#C2C3C5;--shiki-dark:#6B737C">// using normalized time t (in [0,1])</span></span>
<span data-line=""><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">const</span><span style="--shiki-light:#6F42C1;--shiki-dark:#B392F0"> lerp</span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583"> =</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> (start</span><span style="--shiki-light:#212121;--shiki-dark:#BBBBBB">,</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> end</span><span style="--shiki-light:#212121;--shiki-dark:#BBBBBB">,</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> t) </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">=></span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> start </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">+</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> (end </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">-</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> start) </span><span style="--shiki-light:#D32F2F;--shiki-dark:#F97583">*</span><span style="--shiki-light:#24292EFF;--shiki-dark:#B392F0"> t;</span></span>
<span data-line=""> </span></code></pre></figure>
<p>Where <code>t</code> is <code>(currentTime - startTime) / duration</code>.</p>
<p><strong>Easing Functions</strong></p>
<p>Easing changes the <em>feel</em> of an animation (makes it less linear).</p>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2010.png" alt="image.png"></p>
<ul>
<li>Example functions:
<ul>
<li><code>easeOut = (t) => t*t</code></li>
<li><code>flip = (t) => 1 - t</code></li>
<li><code>easeIn = flip(easeOut(flip(t)))</code></li>
</ul>
</li>
<li>Use:</li>
</ul>
<pre><code>value = lerp(startValue, endValue, easing(t));

</code></pre>
<hr>
<p><strong>Keyframes</strong></p>
<ul>
<li>A tween is between 2 keyframes.</li>
<li>Keyframing uses a sequence:</li>
</ul>
<pre><code>[
  { time: 0, value: 100 },
  { time: 1000, value: 300 },
  ...
]

</code></pre>
<ul>
<li>Find two keyframes for current time and interpolate.</li>
</ul>
<hr>
<p><strong>Animation in SimpleKit</strong></p>
<p>Use <code>setSKAnimationCallback((time) => { ... })</code> to animate each frame.</p>
<p>Use it to update positions, change colors, and control timing.</p>
<hr>
<p><strong>Animator Class in SimpleKit</strong></p>
<pre><code>export class Animator {
  constructor(
    public startValue: number,
    public endValue: number,
    public duration: number,
    public updateValue: (v: number) => void,
    public easing: EasingFunction = (t) => t
  ) {}
}
</code></pre>
<p>Call <code>update(time)</code> every frame to animate and interpolate value.</p>
<p><strong>Animation Manager Pattern</strong></p>
<p>An animation manager holds multiple animations and updates them all each frame:</p>
<pre><code>manager.update(currentTime);

</code></pre>
<p>Removes finished ones automatically.</p>
<p><strong>System Timers and UI Threading</strong></p>
<ul>
<li>UI toolkits usually run in a <strong>single-threaded</strong> environment.</li>
<li>Animation callbacks (<code>setInterval</code>, <code>requestAnimationFrame</code>) must run on the UI thread.</li>
<li>Don't update the UI from another thread directly—can cause bugs or exceptions.</li>
</ul>
<p><strong>Example Use Case</strong></p>
<ul>
<li>If square is clicked → turns blue for 2s (using timer).</li>
<li>If click is outside → square moves to the click location over 1s (using tweening).</li>
</ul>

<h2>Dispatch</h2>
<p><strong>Dispatch</strong> refers to the routing of an event to the appropriate <strong>widget</strong> or <strong>code</strong> that needs to handle it. Dispatch takes care of who was clicked (which widget) and how should it handle (what function to call) by finding the right widget based on mouse position or focus and sending the event to that widget so it can react</p>
<p>In UI Architecture: to route an event to the appropriate widget/code</p>
<p><strong>UI Event Flow (Pipeline)</strong></p>
<p>The event moves through the following stages:</p>
<ol>
<li><strong>Low-Level Input</strong> (OS/device)</li>
<li><strong>Event Creation</strong> (Toolkit layer)</li>
<li><strong>Event Queue</strong> (Queued in browser)</li>
<li><strong>Dispatch</strong> (Toolkit routes it to widgets)</li>
<li><strong>App Binding</strong> (Your code handles the event)</li>
</ol>
<p><strong>View Hierarchy</strong></p>
<ul>
<li>UI elements are structured as a <strong>tree</strong> with one root.</li>
<li>Containers group child widgets; child order = draw order.</li>
<li><strong>Overlapping</strong> elements require careful dispatching.</li>
</ul>

<p><strong>Dispatch Steps (uses DFS)</strong></p>
<ol>
<li>
<p><strong>Target Selection</strong></p>
<p>Find which widget is under the mouse or has focus (positional vs focus).</p>
</li>
<li>
<p><strong>Route Construction</strong></p>
<p>Build a path from the <strong>root to the target</strong> widget.</p>
<p>builds a list of widgets from the <strong>root of the widget tree</strong> to the <strong>target widget</strong>. It includes all <strong>ancestors</strong> of the target and is used during event propagation (capture and bubble phases).</p>
</li>
<li>
<p><strong>Propagation</strong></p>
<p>Two phases:</p>
<ul>
<li><strong>Capture (DOWN)</strong>: From root to target</li>
<li><strong>Bubble (UP)</strong>: From target back to root</li>
</ul>
</li>
</ol>
<pre><code>function dispatch(me: SKMouseEvent, root: SKElement) {
  const route = buildTargetRoute(me.x, me.y, root);

  const stopPropagation = route.some((el) => el.handleMouseEventCapture(me));
  if (stopPropagation) return;

  route.reverse().some((el) => el.handleMouseEvent(me));
}

</code></pre>
<p><strong>2 Main Type of Dispatch:</strong></p>
<ul>
<li><strong>Positional Dispatch</strong>: Widget under the mouse (e.g. buttons).</li>
<li><strong>Focus Dispatch</strong>: Widget with current focus (e.g. textfields for keyboard input).</li>
</ul>
<p><strong>Event Binding Mechanisms</strong></p>
<p>Global Event Callback (Old Windows)</p>
<p>A single function like <code>WindowProc</code> handles all events.</p>
<p>A widget uses <code>stopPropagation()</code> when it wants to handle an event <strong>without letting other widgets (like its parents) also react</strong>. This is useful when, for example, a button inside a list item is clicked — you want the button to perform its action, but <strong>stop the event from reaching the list</strong>, so the list doesn't also think the item was selected.</p>

<p><strong>2 Phases of Event Propagaton: Capture vs Bubble</strong></p>
<ul>
<li><strong>Capture Phase</strong>: Goes from root to target (DOWN)</li>
<li><strong>Bubble Phase</strong>: Returns from target to root (UP)</li>
<li>Handlers can stop propagation by returning <code>true</code></li>
<li>Having both gives flexibility: one phase can pre-process or block, the other can respond or clean up.</li>
</ul>

<p>Used for hover effects.</p>

<h2>Layout</h2>
<ul>
<li>a visual arrangement of widgets in a user interface</li>
<li>the way widgets are arranged in the interface (how buttons, text fields, containers, etc are positioned and sized on the screen</li>
</ul>
<p><strong>Box Model:</strong></p>
<ul>
<li>UI elements typically use a hierarchy of dimensions for size</li>
<li>4 dimensions are:
<ul>
<li><strong>margin</strong>: “outside” space away from the other elements</li>
<li><strong>border</strong>: thickness of stroke outlining element</li>
<li><strong>padding</strong>: inside space between the border and the content</li>
<li><strong>content</strong>: the actual content of the element</li>
<li>in SimpleKit, there is no border but we use alternative box model: width = content + padding</li>
</ul>
</li>
</ul>
<p><strong>Widget Sizes:</strong></p>
<ul>
<li><strong>Content Size</strong>: The size of the widget’s main content</li>
<li><strong>Intrinsic Size:</strong> content + padding + margin</li>
<li><strong>layout size</strong>: how much space it can actually take</li>
</ul>
<p><strong>SimpleKit Box Model Implementation</strong></p>
<ul>
<li>SKElement:
<ul>
<li><strong>measure:</strong> calculates intrinsic size using boxmodel</li>
<li><strong>layout</strong> sets element size (intrinsic size or specific size)</li>
</ul>
</li>
<li>SKContainer
<ul>
<li><strong>measure:</strong> calculates intrinsic size with or without children</li>
<li><strong>layout</strong> sets element size (with or without children)</li>
</ul>
</li>
</ul>
<p><strong>2 Types of Layout (Two Pass Layout)</strong></p>
<ol>
<li><strong>Measure</strong> – All widgets calculate their <em>intrinsic size</em>.</li>
<li><strong>Layout</strong> – All widgets are <em>positioned and sized</em> based on layout strategy.</li>
</ol>

<p><code>fillWidth</code> and <code>fillHeight</code> are properties that tell the layout system to stretch the widget to fill the available space horizontally and vertically inside the container.</p>
<p><strong>Layout Implementations</strong></p>
<p>FixedLayout</p>
<p>Widgets manually set their <code>x</code>, <code>y</code>, <code>width</code>, <code>height</code>.</p>
<pre><code>root.layoutMethod = new Layout.FixedLayout();

</code></pre>
<p>CentredLayout</p>
<p>All widgets stack on top of each other, centered in the parent container.</p>
<pre><code>root.layoutMethod = new Layout.CentredLayout();

</code></pre>
<p>WrapRowLayout</p>
<p>Widgets are placed in rows, wrapping when space runs out.</p>
<pre><code>root.layoutMethod = new Layout.WrapRowLayout({ gap: 10 });

</code></pre>
<p>FillRowLayout</p>
<p>Widgets are resized proportionally to fill the container row.</p>
<pre><code>root.layoutMethod = new Layout.FillRowLayout({ gap: 10 });
widget.fillWidth = 1; // Proportional size

</code></pre>
<p><strong>Layout Invalidation</strong></p>
<p>When any size/layout property changes, layout needs to be redone.</p>
<p>To prevent multiple recalculations per frame, SimpleKit uses a flag:</p>
<pre><code>let layoutRequested = false;
function invalidateLayout() {
  layoutRequested = true;
}

</code></pre>
<p>Only one layout update runs per animation frame.</p>
<h2>Model View Controller (MVC)</h2>
<ul>
<li>became a standard design pattern for GUIs</li>
<li>used at many levels like overall application design and individual components</li>
<li>was the first MV* interactive system architectures</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2015.png" alt="image.png"></p>
<ul>
<li>why do we use it?</li>
</ul>
<ol>
<li>Separate data, state, and "business logic" from user-interface
<ul>
<li>Ideally, View and Controller implementations can change without
changing Model implementation, e.g.:</li>
<li>Add support for a new interface (e.g. different device)</li>
<li>Add support for a new input device (e.g., touchscreen)</li>
</ul>
</li>
<li>Supports multiple views of same data, e.g.
<ul>
<li>View numeric data as a table, a line graph, a pie chart, …</li>
<li>Present simultaneous “overview” and “detail” views</li>
<li>Distinct “edit” and “preview” views</li>
</ul>
</li>
<li>Separation of concerns in code
<ul>
<li>code reuse</li>
<li>unit testing</li>
</ul>
</li>
</ol>
<p><strong>MVC Implementation</strong></p>
<p>Interface architecture decomposed into three parts:</p>
<ul>
<li>model: manages application data and logic</li>
<li>view: manages interface to present data</li>
<li>controller: manages interaction to modify data</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2016.png" alt="image.png"></p>
<ol>
<li><strong>Model</strong> stores the data and has logic to update it</li>
<li>When the Model changes, it <strong>notifies</strong> Views (via Observer pattern)</li>
<li><strong>View</strong> then refreshes its display</li>
<li><strong>Controller</strong> translates user input into model updates</li>
</ol>

<h2><strong>Visual Design</strong></h2>
<p>Visual design focuses on how your UI looks and feels. It helps users navigate interfaces more easily and enjoy the experience.</p>
<p><strong>Design Goals</strong></p>
<ul>
<li><strong>Usability</strong>: Easy and intuitive to use</li>
<li><strong>Efficiency</strong>: Reduces user effort and thinking</li>
<li><strong>Clarity</strong>: Clean layout with clear functionality</li>
<li><strong>Aesthetic</strong>: Visually appealing design</li>
<li><strong>Accessibility</strong>: Usable for people with different needs</li>
</ul>
<p><strong>Core Principles of Visual Design</strong></p>
<ul>
<li><strong>Visual Hierarchy:</strong> Directs user attention to the most important parts using size, color, alignment, and spacing.</li>
<li><strong>Alignment:</strong> Elements should line up with each other or with a grid. This keeps the layout tidy and organized.</li>
<li><strong>Proximity:</strong> Related items should be placed close together. Helps users understand which elements belong together.</li>
<li><strong>Contrast:</strong> Using strong differences (like bold text or color) to highlight or emphasize important parts.</li>
<li><strong>Repetition:</strong> Repeating styles like fonts or button shapes helps make the interface feel consistent and easier to learn.</li>
<li><strong>Balance:</strong> Visually distributes elements so the layout doesn’t feel heavy on one side. This can be symmetrical or asymmetrical.</li>
<li><strong>White Space:</strong> Empty space between or around elements. It improves focus, readability, and prevents a cluttered look.</li>
</ul>
<p><strong>Typography Tips</strong></p>
<ul>
<li>Use 1 or 2 font families</li>
<li>Use font size to create structure (headings vs body)</li>
<li>Adjust spacing and line-height for better legibility</li>
</ul>
<p><strong>Color Use</strong></p>
<ul>
<li>Pick a small color palette (3–4 key colors)</li>
<li>Use color intentionally (e.g., to show status or draw attention)</li>
<li>Make sure there’s enough contrast between text and background</li>
<li>Use tools like Adobe Color or Coolors for accessible palettes</li>
</ul>
<p><strong>Icons and Imagery</strong></p>
<ul>
<li>Use simple, recognizable icons</li>
<li>Icons should match in style and size</li>
<li>Avoid random decorative images without meaning</li>
<li>Always pair icons with text if clarity is needed</li>
</ul>
<p><strong>Grid Systems</strong></p>
<ul>
<li>Use columns and consistent spacing</li>
<li>Helps with alignment and scaling across screen sizes</li>
<li>Common choice: 12-column grid layout</li>
<li>uses <code>@media</code> query to change grid layout (mobile responsivness)</li>
</ul>
<p><strong>Accessibility Considerations</strong></p>
<ul>
<li>Ensure high contrast between text and background</li>
<li>Let users resize text</li>
<li>Avoid using color alone to indicate state</li>
<li>Add alternative text to images for screen readers</li>
</ul>
<p><strong>Typeface vs Font</strong></p>
<ul>
<li>Typeface = style of text design</li>
<li>Font = specific implementation of a typeface</li>
</ul>
<p>A <strong>Design System</strong> is a systematic approach to product development, complete with guidelines, principles, philosophies, and code</p>
<p><strong>Primary Colors:</strong> main brand/interface, usually detailed in the layout, key elements, text, etc</p>
<p><strong>Secondary Colors:</strong> used for functional meaning such as alert</p>
<h2>Text</h2>
<p><strong>Text Representation and Encoding</strong></p>
<p>Text is not stored as characters but as numbers. These numbers are known as code points.</p>
<ul>
<li><strong>ASCII</strong>: An early standard using 7 bits, representing 128 characters. Only supports English.</li>
<li><strong>Unicode</strong>: A modern character set supporting most global languages. Each character has a unique number (code point).</li>
<li><strong>UTF-8</strong>: The most common encoding on the web. It’s backward-compatible with ASCII and uses 1 to 4 bytes per character depending on the symbol.</li>
</ul>
<p><strong>Internationalization (i18n) vs Localization (l10n)</strong></p>
<ul>
<li><strong>i18n (Internationalization)</strong>: Designing software in a way that allows it to be easily adapted for various regions and languages. It's about enabling flexibility.</li>
<li><strong>l10n (Localization)</strong>: Actually translating and adjusting the content for a specific locale, such as language, date formats, currency, etc.</li>
</ul>
<p><strong>HTML Forms</strong></p>
<ul>
<li>: Wraps input elements.</li>
<li>: Lets users enter text or other data.</li>
<li>: Describes the input field.</li>
<li>: Submits or triggers an action.</li>
</ul>
<p><strong>Input Attributes:</strong></p>
<ul>
<li><code>type</code>: Sets the kind of data expected (e.g., "text", "email", "password")</li>
<li><code>placeholder</code>: Shows a hint inside the input</li>
<li><code>required</code>: Prevents submission if the field is empty</li>
<li><code>maxlength</code>: Limits number of characters</li>
<li><code>pattern</code>: Validates input against a regular expression</li>
</ul>
<p><strong>Client-Side Validation</strong></p>
<p>Handled in the browser before the form is submitted. Helps prevent incorrect or missing data from reaching the server.</p>

<p><strong>Input Formatting vs Input Masking</strong></p>
<ul>
<li><strong>Formatting</strong>: Adjusts the input after it’s entered. Example: showing (519) 555-1234 after submitting a plain number.</li>
<li><strong>Masking</strong>: Adds format as the user types. Example: formatting credit card or phone numbers on the fly.</li>
</ul>
<p><strong>Why It Matters:</strong> Both help improve readability and user experience while reducing errors.</p>
<p><strong>Input Sanitization</strong></p>
<p>Users can input harmful data. If not sanitized, this can lead to:</p>
<ul>
<li>Cross-site scripting (XSS)</li>
<li>SQL injection</li>
<li>Application crashes or corrupted databases</li>
</ul>
<p><strong>Sanitize both client-side and server-side:</strong></p>
<ul>
<li>Client-side sanitization improves usability.</li>
<li>Server-side sanitization protects the system.</li>
</ul>
<h2>Undo</h2>
<p>4 Design Choices:</p>
<ul>
<li>Undoable Actions
<ul>
<li>some actions may be omitted from undo</li>
<li>some actions are destructive and not easy undone</li>
<li>destructive actions may need confirmation</li>
</ul>
</li>
<li>State Restoration
<ul>
<li>what will the UI look like after an undo or redo?</li>
<li>UI State should be meaningful after redo action</li>
</ul>
</li>
<li>Granularity
<ul>
<li>how much should be undone at one time</li>
<li>how big each undo step is (<strong>chunk)</strong>
<ul>
<li>one chunk = one logical change</li>
</ul>
</li>
</ul>
</li>
<li>Scope
<ul>
<li>is undo global or for a specific part?</li>
</ul>
</li>
</ul>
<p>Implementation Approaches:</p>
<ul>
<li><strong>Forward Undo</strong>
<ul>
<li>start from base state and list of changes</li>
<li>to undo, skip the last change when rebuilding</li>
<li><strong>Disadvantage:</strong> Can use a lot of memory if storing full state for each step, especially for large documents or images.</li>
</ul>
</li>
<li><strong>Reverse Undo</strong>
<ul>
<li>Apply change immediately and store the reverse change.</li>
<li>To undo: apply the reverse change.</li>
<li>would be good to implement if dealing with destructive actions</li>
<li><strong>Redo invalidation:</strong> After a sequence of undos, what user action <strong>kills the redo stack</strong>, and why is that the right behavior?
<ul>
<li><strong>Rule:</strong> Any <em>new</em> user action after an undo clears the redo stack.</li>
<li><strong>Why:</strong> Redo history only makes sense if you continue along the same branch of changes; a new edit creates a new branch, so the old redo chain is no longer valid.</li>
</ul>
</li>
</ul>
</li>
</ul>
<p><strong>Implementation with Stacks</strong></p>
<p>Using either of these options requires two stacks</p>
<ul>
<li><strong>Undo stack</strong>: all change records, saved as you perform actions</li>
<li><strong>Redo stack</strong>: change records that have been "undone" (needed to reapply them with redo)</li>
</ul>
<p><strong>Memento</strong></p>
<ul>
<li>simple example of undo</li>
<li>essentially saves the snapshots of states so it would essentially be the screenshots or something</li>
<li>the downside is that it uses more memory but it is simple approach</li>
</ul>
<h2>Direct Manipulation</h2>
<ul>
<li>virtual representation of an object that is manipulated in a similar way to a real-world object</li>
<li>make the interaction feel like manipulating a ‘real object’ instead of working through an intermediary</li>
<li>requires a representation of ‘task objects’ (something a user can manipulate)</li>
</ul>
<p>| Real World | Interface |
| --- | --- |
| Move hand to object | Move pointer to object |
| Pick up with hand | Click and hold |
| Carry to trash bin | Drag to trash icon |
| Release | Release mouse button |</p>
<p>Benefits:</p>
<ul>
<li>feels like you are interacting with objects, not a computer</li>
<li>focus stays on <strong>task</strong> not on figuring out the interface</li>
<li>gives a sense of control and engagement</li>
</ul>
<p><strong>Dragging</strong></p>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2018.png" alt="image.png"></p>
<ul>
<li>mousedown on shape starts draft, calculate <em>offset</em> from position to the shape frame-of-reference</li>
<li>mousemove to drag</li>
<li>mouseup to end drag</li>
</ul>
<p><strong>Transformable Shape:</strong></p>
<ul>
<li>direct manipulation widgets to transform an object:
<ul>
<li>scale by dragging corners and side handles</li>
<li>rotate by dragging the rotation handle</li>
<li>translate by dragging shape</li>
</ul>
</li>
</ul>
<h2>Instrumental-Interaction</h2>
<ul>
<li>a model of interaction based on how we naturally use tools or instruments to manipulate objects in the physical world
<ul>
<li>interfaces have interaction instruments and domain objects</li>
</ul>
</li>
</ul>
<p><strong>3 Properties to describing instruments</strong></p>
<ul>
<li><strong>Degree of Indirection</strong>
<ul>
<li>two dimensional measure of ‘distance’ from instrument to object</li>
<li>spatial dimension
<ul>
<li><em>near:</em> drag to translate, handle to resize</li>
<li><em>medium:</em> scrollbar near page</li>
<li><em>far:</em> dialog box</li>
</ul>
</li>
<li>temporal dimension
<ul>
<li><em>short:</em> direct drag response like drag shape</li>
<li><em>med</em>: activate tool in toolbar, start direct manipulation</li>
<li><em>long:</em> using dialog, full drag-and-drop operation</li>
</ul>
</li>
</ul>
</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2019.png" alt="image.png"></p>
<ul>
<li>
<p><strong>Degree of Integration</strong></p>
<ul>
<li>ratio of degree of freedom of the instrument over the DOF captured by input device</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2020.png" alt="image.png"></p>
</li>
<li>
<p><strong>Degree of Compatibility</strong></p>
<ul>
<li>Similarity of physical actions on instrument and response of the object</li>
<li>more similar is better since it’s more natural or intuitive</li>
</ul>
</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2021.png" alt="image.png"></p>
<p>| Feature | <strong>Direct Manipulation</strong> | <strong>Instrumental Interaction</strong> |
| --- | --- | --- |
| <strong>Interaction Style</strong> | Directly interact with the object itself | Use a separate tool (instrument) to act on the object |
| <strong>Pros</strong> | - Feels intuitive and natural- Immediate feedback- Easy to learn | - More flexible and powerful- Reusable instruments- Supports complex actions |
| <strong>Cons</strong> | - Limited to simple tasks- Can clutter the UI- Not reusable | - Higher learning curve- More indirect- Visual/tool complexity |
| <strong>Example</strong> | Dragging a shape to move it | Using a slider to resize an object |
| <strong>Best For</strong> | Simple, direct interactions | Advanced, multi-step, or customizable interactions |</p>
<p><strong>Summary:</strong></p>
<ul>
<li>
<p><strong>Domain Object</strong>:</p>
<p>The <strong>thing you want to change</strong>, like a shape on a canvas or a file in a folder.</p>
</li>
<li>
<p><strong>Instrument Object</strong>:</p>
<p>The <strong>tool you use</strong> to change the domain object (e.g., mouse, slider, toolbar button). Think of it like using a paintbrush instead of painting with your hands.</p>
</li>
<li>
<p><strong>Indirection</strong>:</p>
<p>How <strong>far the tool is</strong> from the object:</p>
<ul>
<li>Low = dragging/resizing directly</li>
<li>Medium = using a scrollbar</li>
<li>High = editing with a dialog box</li>
</ul>
</li>
<li>
<p><strong>Reification</strong>:</p>
<p>Turning a <strong>tool or action into a visible object</strong> so it can be reused (e.g., having a slider for volume control that can be moved, copied, or reset).</p>
</li>
</ul>
<p>✅ Pros:</p>
<ul>
<li>Better <strong>control and precision</strong></li>
<li>Tools can be <strong>reused</strong></li>
<li>Good for <strong>complex tasks</strong></li>
</ul>
<p>⚠️ Cons:</p>
<ul>
<li>Can feel <strong>less direct</strong></li>
<li>Slightly more <strong>complex to learn</strong></li>
<li>May <strong>use more memory</strong> if not optimized</li>
</ul>
<h2>Asynchronous</h2>
<p><strong>System performance</strong> measures how fast the system <em>actually</em> runs tasks, while</p>
<p><strong>perceived responsiveness</strong> is how fast the user <em>feels</em> the system is responding — even if it’s still processing in the background.</p>
<p><strong>Event Loop:</strong>  The event loop checks continuously if there are tasks waiting in the task queue or microtask queue and pushes them to the call stack when it's empty. This allows the UI to stay responsive by deferring slow or blocking operations to be handled later.</p>
<p>A <strong>responsive UI</strong> <em>feels</em> like it responds in a timely manner</p>
<p>it is usually accomplished in two ways:</p>
<ul>
<li>designing for human perception and expectations</li>
<li>using asynchronous execution (so the interface is not frozen while something else is happening</li>
</ul>
<p><strong>continuous latency</strong></p>
<ul>
<li>minimal time to be affected by a visual stimulus: max is 10ms where continuous input latency should be less than 10ms</li>
<li>why it matters?
<ul>
<li>even a small delay makes the interaction feel unresponsive</li>
<li>the user’s mental model expects the system to respond instanly, like moving a real object</li>
<li>examples:
<ul>
<li>low latency (good): in drawing app, stroke appears exactly where  the cursor/pen moves</li>
<li>high latency (bad bad): the stroke lags behind, making it harder to draw accurately</li>
</ul>
</li>
</ul>
</li>
</ul>
<p><strong>input feedback</strong></p>
<ul>
<li>immediate acknowledgement that the system has recieved and understood the user’s input via visual, auditory or haptic</li>
<li>it matters becasue user may think their input was missed, double-click etc</li>
<li>this will set a expectation of what will happen next</li>
<li>example
<ul>
<li>button highlight when pressed</li>
<li>slider updates value while being dragged</li>
</ul>
</li>
<li>maximum interval between cause-effects: 140 ms
<ul>
<li>eg. time between pressing a button until the feedback changes</li>
</ul>
</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2022.png" alt="image.png"></p>
<p><strong>graceful degradation of feedback</strong></p>
<ul>
<li>simplify feedback for high-computation continuous input tasks
<ul>
<li>example is window manager updates window rendering after drag. graphics editor only draws object outlines during manipulation</li>
</ul>
<img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2023.png" alt="image.png"></li>
</ul>
<p><strong>busy indicators</strong></p>
<ul>
<li>display busy indicators for operations that take 1s to about 3-4s</li>
</ul>
<p><strong>progress bars:</strong></p>
<ul>
<li>display progress bars for operations more than 3-4s</li>
<li>show work remaining, not work completed</li>
<li>use human precision not computer precision (bad: 333.5 seconds remaining ; good: about 4 minutes)</li>
<li>smooth progress rather than erratic burts</li>
</ul>
<p><strong>progressive loading</strong></p>
<ul>
<li>use skeleton placeholders when loading takes more than 1s</li>
<li>user adjusts to a layout which they will see</li>
<li>loading process seems faster because there is an initial result</li>
<li>provide user with <em>some</em> data while loading rest of data</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2024.png" alt="image.png"></p>
<p><strong>asynchronous execution:</strong></p>
<ul>
<li>execute tasks independently from the main program flow(”do more than one thing at once”)</li>
<li>two main types of asynchronous execution:
<ul>
<li>asynchronous programming
<ul>
<li><strong>Asynchronous programming</strong> runs tasks without blocking the main thread, but still on a single thread — tasks are queued and run when the thread is free (e.g., JS event loop).</li>
<li>singe thread, event driven</li>
<li>related concepts in JavaScript and other languages
<ul>
<li>event driven programming</li>
<li>promise/future</li>
<li>coroutines</li>
<li>non-blocking I/O</li>
</ul>
</li>
<li>call backs (functions triggered later)</li>
<li>async/await → cleaner syntax for working with promises</li>
</ul>
</li>
<li>threading
<ul>
<li>manage multiple concurrent threads with shared resources but executing different instructions</li>
<li>threads are a way to divide computation, reduce blocking</li>
<li>concurrency risks: eg two threads update a variable</li>
</ul>
</li>
</ul>
</li>
</ul>
<p><strong>Prediction Strategy:</strong> use idle time to pre-compute or pre-load the next actions that is likely so they happen instantly when requested.</p>
<ul>
<li>example is in a search app, while the user is viewing the current search result, start pre-loading the next result in the background so it opens instantly if they click “next”</li>
</ul>
<p><strong>Web Worker:</strong></p>
<ul>
<li>kinda like a helper running in the background of your browser</li>
<li>Definition: a background script in the browseer that runs tasks without blocking the main UI thread, communicating with the main script via messages</li>
<li>does the heavy work without freezing your main page</li>
<li>worker cant touch the page directly, can only send messages back and forth to the main script</li>
<li>can talk to it with <code>postMessage()</code> and listen with <code>onmessage</code></li>
</ul>
<p>| Feature | <strong>Normal Async Execution (Promises, fetch)</strong> | <strong>Web Worker</strong> |
| --- | --- | --- |
| <strong>Threading</strong> | Single-threaded (still runs on the main thread, just delayed until call stack is clear) | Runs in a <strong>separate thread</strong> from the main UI |
| <strong>Purpose</strong> | Non-blocking I/O tasks (network requests, timers) | Offload <strong>heavy computation</strong> so the UI remains responsive |
| <strong>Example</strong> | <code>fetch()</code>, <code>setTimeout()</code> | Image filtering, parsing large datasets |
| <strong>UI Access</strong> | Can directly update DOM | <strong>Cannot</strong> touch DOM (must send messages back) |
| <strong>Execution Model</strong> | Uses <strong>event loop</strong> + <strong>task/microtask queues</strong> | Runs independently and communicates via <strong>message passing</strong> |</p>
<h2>Declarative</h2>
<p><strong>Imperative Programming</strong></p>
<ul>
<li>describe how to achieve a result</li>
<li>SimpleKit used an imperative paradigm to build a UI. Manually creates and arranges elements</li>
</ul>
<p><strong>Declarative Programming</strong></p>
<ul>
<li>describe what result you want</li>
<li>HTML is declarative as it describes the final structure of the DOM tree</li>
<li>goal is still to create a tree of nodes with events but you <strong>describe</strong> the tree instead of building it step by step</li>
<li>state changes causes full re-renders of the UI (HTML,JSX,HyperScript</li>

<p><strong>Preact</strong></p>
<ul>
<li>render quickly and efficiently</li>
<li>DOM (document object model) is a lightweight JS object of the UI.</li>
<li>differences between react and preact is in preact, events use the <em>browser’s native DOM events</em>, while React uses its own <em>synthetic event system</em></li>
<li>preact uses native DOM events</li>
<li>virtual DOM only updates the changed parts of the real DOM → faster</li>
<li>effective memory usage</li>
<li>treats children nodes as native javascript arrays (basically uses an array instead of having children</li>
<li>uses class instead of className in React</li>
<li>we use one root node becasue JSX compiles into a tree structure and that tree needs a single top-level element to attach to the DOM</li>
<li>components are the building blocks of a preact application where they have custom props</li>
<li>props: data passed from parent to child where child is read-only and can be used for customization</li>
<li>hooks: functions like <code>useState</code>, <code>useEffect</code>, <code>useSignal</code> and it should only be called at top-level, same order every render, only inside components</li>
</ul>

<h2>Reactivity</h2>
<ul>
<li>can be defined as the ‘automatic’ update of the UI due to a change in the application’s state</li>
<li>The <strong>Virtual DOM</strong> is an <strong>in-memory representation of the real DOM</strong>. Frameworks use it to <strong>batch and minimize real DOM updates</strong>, making UI rendering faster and more efficient because direct DOM manipulation is expensive.</li>
<li>VDOM Reconciliation
<ul>
<li>save copy of current VDOM</li>
<li>components/application state updates the VDOM</li>
<li>re-render is triggered by framework</li>
<li>compare VDOM <em>before update</em> with VDOM <em>after update</em></li>
<li>reconcile the difference by identifying a set of DOM patches</li>
<li>perform patch operations on real DOM</li>
</ul>
</li>
<li>if node type changes, the whole subtree is rebuilt</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2026.png" alt="image.png"></p>
<ul>
<li>if node type is the same, attributes are compared and updated</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2027.png" alt="image.png"></p>
<ul>
<li>if a node is inserted into list of same node type siblings, all children will be updated</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2028.png" alt="image.png"></p>
<ul>
<li>When updating children of same node type, use key prop
<ul>
<li>each key must be stable and unique</li>
<li>key should be assigned when data created, not when rendered</li>
</ul>
</li>
</ul>

<p>Approach works best with declarative UI programming</p>
<ul>
<li>e.g. components in Preact</li>
</ul>
</li>
<li>
<p><code>@apply</code> let you use <strong>utility classes</strong> inside your own CSS so you can group styles and avoid repeating class lists in HTML</p>
</li>
</ul>
<h2>Accessibility</h2>
<p>Definition: developing content to be as accessible as possible, no matter an individual’s physical and cognitive abilities and how they access a user interface. often abbreviated as <code>a11y</code></p>
<p>Access to User Interface is considered as <em>human right</em></p>
<p><strong>Abilities</strong></p>
<ul>
<li>People vary in physical and mental capabilities
<ul>
<li>How are your abilities different from other people?</li>
<li>How will your abilities change in the future?</li>
<li>How will your abilities change in different environments</li>
</ul>
</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2030.png" alt="image.png"></p>
<p><strong>Types of Long-term Impairments</strong></p>
<ul>
<li><strong>Visual</strong>: blindness, low vision.</li>
<li><strong>Hearing</strong>: deafness, partial hearing loss.</li>
<li><strong>Motor</strong>: wheelchair use, limited hand dexterity.</li>
<li><strong>Cognitive</strong>: memory loss, learning disabilities.</li>
<li>Aging increases chances of impairment.</li>
</ul>
<p><strong>OS Accessibility Features</strong></p>
<ul>
<li>Zoom, screen readers, captions, speech dictation, keyboard cursor control, voice input.</li>
</ul>
<p><strong>Enhancements</strong></p>
<ul>
<li><strong>Visual</strong>: larger text, high contrast, dark mode, remove animations, screen readers.</li>
<li><strong>Hearing</strong>: visual alerts, sound recognition, background noise filtering.</li>
<li><strong>Motor</strong>: sticky keys, eye tracking, physical switches.</li>
<li><strong>Cognitive</strong>: text-to-speech, icons, slower interfaces, reduced animations.</li>
</ul>
<p><strong>Curb Cut Effect:</strong> Accessibility features often help everyone (e.g., captions help in noisy areas, learning languages).</p>
<p><strong>Legal Requirements</strong></p>
<ul>
<li>Canada: Accessible Canada Act (federal), AODA (Ontario) → must follow WCAG (Web Content Accessibility Guidelines) standards.</li>
<li>US: Section 508, ADA.</li>
<li>EU: Web Accessibility Directive.</li>
</ul>
<p><strong>Four WCAG Principles:</strong></p>
<ul>
<li><strong>Perceivable</strong> – Information must be presented so users can perceive it. Example: Provide alt text for images.</li>
<li><strong>Operable</strong> – Users must be able to interact with the interface. Example: Ensure all controls are usable via keyboard.</li>
<li><strong>Understandable</strong> – Content and operation should be clear. Example: Use simple, consistent navigation.</li>
<li><strong>Robust</strong> – Content should work with different assistive tech. Example: Use semantic HTML so screen readers can interpret structure.</li>
</ul>
<p><strong>Implementing Accessibility</strong></p>
<ul>
<li><strong>Accessibility Tree</strong>: browser’s version of the DOM for assistive tech.</li>
<li><strong>The Accessibility Tree</strong> is a version of the DOM that the browser creates for assistive technologies like screen readers. It contains only the information these tools need, such as roles, names, and states of elements.</li>
<li>Use <strong>Semantic HTML</strong> (headings, lists, nav, section) instead of just <code>&#x3C;div></code> and <code>&#x3C;span></code>.</li>
</ul>
<p><img src="CS%20349%20User%20Interfaces%201d941046f5a680f386cdc6fac01076da/image%2031.png" alt="image.png"></p>
<ul>
<li>Skip Links: let users jump straight to main content.</li>
<li><strong>ARIA</strong>: add extra meaning for assistive tech (only if semantic HTML can’t do it).</li>
<li><strong>Accessible forms</strong>: proper labels, legends, and fieldsets.</li>
<li>Colour contrast: WCAG (Website Content Accessibility Guidelines) min = 4.5:1 (text vs background)
<ul>
<li>The WCAG minimum contrast ratio for normal text is <strong>4.5:1</strong>. This ensures that text is readable for users with low vision or in poor lighting conditions.</li>
</ul>
</li>
</ul>
